Nodus

A graph-based puzzle game where every arrow is rigid, every exit is deliberate.

Overview AI Workflow Technical Highlights Implementation Closing

01 Project Overview

Each level is a graph of nodes and edges covered by rigid arrows. Tapping an arrow attempts one atomic exit in its head direction: it either escapes the board entirely, or the move is rolled back. There is no partial movement, so every level is a question of order — which arrow can leave now so that the others eventually can too.

Graph-Based Mechanics

The board is a graph, not a grid of tiles: nodes joined by edges, with rigid multi-node arrows laid over them. Movement is a coordinate sweep from the arrow's head — clear path, the whole arrow slides off; blocked, the attempt is undone atomically.

2D & 3D Modes

Flat 2D boards and multi-layer 3D boards viewed through an orbitable perspective camera. Both run on the same resolver — the extra dimension is a direction abstraction, not special-cased physics.

Challenge System

Three challenge types reuse campaign levels under new pressure: Time Attack, Move Limit, and Perfect Run. Their results live in separate storage and never touch campaign completion, progress sync, or the leaderboard.

Progressive Difficulty

30 levels: 15 generated rectangular boards, 5 figure silhouettes (heart, diamond, club, spade, crown), and 10 multi-layer 3D figures. Difficulty is computed from each level's structure — arrow count, blocked starts, bends, density, layers — and the level list is ordered by that score.

Audio System

An app-lifetime audio manager keeps one set of players alive across screens instead of rebuilding them per level. Music and effects negotiate focus so they duck rather than cut each other, and pause and resume with the app.

Progress & Leaderboards

Progress is local-first: unlocks and best scores live on the device, so the game is fully playable offline. An optional account adds cloud sync that never discards a better local result, plus per-level leaderboards when signed in.

02 AI Workflow Evolution

Nodus was built with an AI coding assistant, and the way the team worked with it changed as much as the game did. Every request fills a fixed-size context window — the text the model reads before it answers. The three stages below are about spending less of that window re-explaining the project, and more of it producing correct work.

1

Stage 1

Prompt Engineering

Carefully worded one-off prompts. Each session re-explained the architecture, the constraints, and the history from scratch, inside the prompt itself.

Why the project moved on Context was rebuilt in full every time. Prompts grew unwieldy, facts drifted between sessions, and much of the window went to restating what the project already knew before any real work began.

2

Stage 2

Spec-Driven Development

Work moved to written specifications: structured phase documents stating exactly what to build, which files were in scope, and which rules applied.

Why the project moved on Ambiguity and rework dropped, but each phase still carried a bespoke document duplicating the same baseline facts by hand. The shared knowledge lived nowhere permanent.

3

Stage 3 · current

Harness Engineering

The project keeps a standing harness/ directory: baseline context, active rules, pre- and post-implementation checklists, phase templates, and metrics. A new phase references that shared state instead of restating it — only the delta enters the window. The cost is upkeep: a changed decision must be edited there once, or future phases read a stale fact.

Why it stuck The baseline is written once and reused everywhere. Facts stay consistent, mistakes are logged rather than repeated, and the window is freed for implementation and review. This is the workflow the project runs on today.

03 Technical Highlights

Not just what the stack is, but why each piece earns its place.

AWS EC2 Hosting

The backend runs on an EC2 instance — a real, always-on server with full control over runtime and network, and a chance to operate cloud infrastructure directly rather than through a managed sandbox.

Docker-Based Deployment

A Dockerfile and docker-compose.yml start the API and its database with one command. The same image runs on a laptop and on EC2, removing "works on my machine" drift.

Local / Cloud Flexibility

The app is fully playable offline against local level data. Auth, cloud sync, and leaderboards are strictly additive — a network outage never blocks play.

Flutter + Clean Architecture

One Flutter codebase across platforms, organised Domain → Application → Infrastructure → Presentation per feature. A pure domain (no Flutter, HTTP, or storage imports) lets the game rules be unit-tested in isolation and reused unchanged across 2D and 3D.

Node.js + NestJS + Prisma

A NestJS API whose modular, dependency-injected structure mirrors the frontend's layering, with Prisma as the type-safe ORM and migration layer — a strongly-typed path from HTTP request to database and back.

Graph-Based Game Engine

No matrix, grid-cell, or tile model exists in the runtime. Levels are graphs; movement is a coordinate sweep resolved entirely in the domain layer. That one abstraction drives flat boards, figure silhouettes, and stacked 3D levels alike.

30 Levels, Tool-Validated

15 generated rectangular boards, 5 figure silhouettes, and 10 multi-layer 3D figures. A Node generator/validator checks every level for solvability, connectivity, and shape rules before it ships.

Challenge Modes

Time Attack, Move Limit, and Perfect Run layer over campaign levels via a scoring strategy pattern, with fully separate save state. A new challenge type means one new strategy — the core loop stays untouched.

Backend-Driven Dynamic Levels

The client can fetch extra playable levels from the backend (reserved numbers 1000+) and merge them in at runtime, so new content ships by seeding the database instead of rebuilding the app. The path is implemented end-to-end but still gated behind the ENABLE_REMOTE_LEVELS build flag, off by default. When on, the merge is offline-first: the 30 local levels stay authoritative and win any number conflict, and an absent backend falls back to the last cached remote batch.

04 Technical Implementation

What each side of the stack owns, how the two relate, and the cross-cutting concerns applied to the backend.

Frontend Responsibility

The Flutter client owns all gameplay: the graph domain, movement resolution, rendering, audio, and local persistence. The domain layer stays pure Dart, so the rules deciding whether an arrow escapes or collides are unit-tested with no UI and no network.

Backend Responsibility

The NestJS API owns accounts, the level catalog, progress sync, and leaderboards — organised as src/{domain, application, infrastructure, interfaces}, with Prisma and Swagger docs at /api/docs. It never runs gameplay logic; that stays on the client.

How Frontend and Backend Relate

Strictly additive. Local levels are the offline source of truth; the client maps them to backend ids via GET /levels. Progress sync never discards a better local result, and leaderboard submission — authenticated only — is best-effort and non-blocking, so an absent backend never blocks local play.

Architecture Impact

Both repositories use Clean/hexagonal architecture. Business rules isolated from frameworks can be tested without booting a UI or database; adapters (Prisma repository, HTTP client, SharedPreferences) swap behind a port without touching a use case; and an adapter failure can't leak into the rules that decide correctness.

AOP — Three Cross-Cutting Concerns

NestJS interceptors, filters, and guards keep cross-cutting concerns out of business logic. Two are global: a logging & performance interceptor wraps every handler, recording method/path/status/time with zero controller changes; an exception filter normalises any thrown error into one JSON shape. Security is per-route: a JWT guard protects progress, leaderboard submission, and admin endpoints, and a roles guard with @Roles(ADMIN) further restricts the admin-only level endpoints.

Walkthrough

Request Lifecycle: From Tap to Response

Submitting a leaderboard score — every layer the request crosses on both sides, and where each AOP aspect (cross-cutting concern) attaches along the way.

  1. 1
    Player action → domain use case Frontend

    Completing a level runs MoveArrowUseCase/GameSessionService entirely on-device. The result goes to SyncProgressUseCase, which depends only on the RemoteProgressRepository port — not on HTTP.

  2. 2
    Outbound adapter wraps the call Frontend

    HttpApiClient — the port's implementation — builds the HTTP request.

    Frontend AOP: a TokenProvider callback attaches Authorization: Bearer <token>, so the use case never touches auth headers, and the single _send method wraps every failure into one ApiException shape.

  3. 3
    Request reaches the backend gateway Backend

    POST /leaderboard arrives at leaderboard.controller.ts — the driving adapter, the only entry point the outside world may call.

    Backend AOP: JwtAuthGuard validates the bearer token per-route before the handler runs; LoggingPerformanceInterceptor starts timing globally.

  4. 4
    Controller → use case → port Backend

    The controller validates the DTO and calls the use case, which depends only on the LeaderboardRepository port and LeaderboardScorePolicy — it has no idea Prisma, Postgres, or HTTP exist.

  5. 5
    Driven adapter touches the database Backend

    PrismaLeaderboardRepository implements the port and is the only code that talks to Postgres. Swapping it for an in-memory fake in tests changes nothing above this layer.

  6. 6
    Response leaves through the same gateway Backend

    The result bubbles back to the controller, which returns it as JSON.

    Backend AOP: the global HttpExceptionFilter catches anything thrown and normalises it, so no controller formats its own errors; LoggingPerformanceInterceptor logs METHOD /path STATUS TIMEms on the way out.

  7. 7
    Frontend receives the result Frontend

    HttpApiClient deserialises the response (or maps a failure to ApiException) and SyncProgressUseCase merges it into local state. Submission is best-effort — a failed backend never blocks local progress.

Every use case in this chain calls only an interface/port — never a concrete adapter, never an AOP aspect. The aspects sit entirely at the two gateways (interfaces/ on the backend, HttpApiClient on the frontend), which is exactly why they can be added or changed without touching a use case.

05 Closing

Nodus began as a small graph-based puzzle prototype and grew into a full 2D and 3D game with an optional online backend — alongside a development workflow that matured beside it, from prompt engineering, to specs, to a harness. The game and the way it was made reflect the same idea: write the rules down once, keep them clean, reuse them everywhere.
