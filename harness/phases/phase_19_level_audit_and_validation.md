## Context

Phases 3–18 are complete and merged to `main`. The game now has **20 manual
levels** in `assets/levels/manual_levels.json`:

- **Levels 1–15** — random-partition rectangular/irregular boards.
- **Levels 16–20** — figure silhouettes: heart (16), diamond (17), club (18),
  spade (19), crown (20). Added in **Phase 16**, *after* the collision fixes
  below.

Current baseline: **122 tests passing**, `flutter analyze` clean,
`node tool/gen_levels.js --validate-only` → ALL VALID: true.

**Why this phase exists — the risk.** Two collision/shape bugs were found and
fixed in levels 1–15:

- **Phase 14** — *interior gap exit bug*: hard-level boundary-node removal left
  interior coordinate gaps inside the board's bounding box. An arrow's head
  sweep hitting a gap made `nodeByCoordinate` return `null`; the resolver
  treated the gap as the board edge and reported `escaped`, skipping a visible
  blocker just past the gap. Fixed via `hasInteriorGapExit` / `flipInteriorGapArrows`.
- **Phase 14.1** — *self-intersecting arrows*: a U-spiral arrow (Level 12 `a15`)
  whose head pointed back into its own tail, exiting by sweeping through its own
  body. Fixed via `hasSelfIntersectingArrow` in generator + validator.

**The concern:** Levels 16–20 were generated by `generateFigureLevel`, which by
design **skips `hasInteriorGapExit` / `flipInteriorGapArrows`** (a concave
figure silhouette is simply connected but bbox-relative gap-checking flags ~100%
of valid partitions as false positives — see Phase 16 notes and
`LEVEL_AUTHORING.md §15`). It **does** apply `hasSelfIntersectingArrow`. The
figure levels therefore may **not** carry the Phase 14 gap-exit protection, and
the Phase 17/18 rendering polish (neon colors, coverage-based node alpha, scaled
arrowheads, scaled `hitSlop`, pinch-to-zoom) has only been verified by automated
tests — never audited specifically against the dense figure boards.

**Stay on branch `main`** (latest merged phases through 18). Do **not** switch
branches. If code changes are required, create `feat/phase-19-level-audit`.

## Task

### 1 — Audit levels 1–15 (verify previous fixes still hold)
- Run `node tool/gen_levels.js --validate-only`.
- Confirm `gapExit=-` and no self-intersection (`shared=-`, structure clean) for
  every level 1–15.
- Document any regression against the Phase 14 / 14.1 guarantees.

### 2 — Deep audit levels 16–20 (figure levels)
For **each** of levels 16 (heart), 17 (diamond), 18 (club), 19 (spade),
20 (crown), verify all of the following. Simulate the actual head sweeps against
`assets/levels/manual_levels.json` — do **not** assume the generator's checks
covered them, because `generateFigureLevel` deliberately omits the gap-exit
check.

a. **Interior gap exits** — simulate the coordinate-based head sweep for every
   arrow. Does any arrow's head exit through a coordinate gap *inside* the
   level's bounding box (a `null` node between the head and a visible blocker)?
   This is the exact Phase 14 bug class, and the figure generator does **not**
   guard against it.
b. **Self-intersection** — does any arrow's head sweep hit a node in its own
   body (covered set minus head)? (Phase 14.1 bug class.)
c. **Arrow shape validity** — is every arrow a simple path? No cycles, no
   branches (interior nodes degree 2, endpoints degree 1), head direction
   consistent (the single body edge at `endNodeId` leads opposite `direction`)?
d. **Node/edge disjointness** — do any two arrows share a node or an edge?
e. **No-free-nodes** — is every visible node occupied by some arrow at level
   start?
f. **Solvability** — does the greedy solver complete the level?
g. **Density** — is the arrow count within the hard-tier band (20–50)?
h. **Direction variety** — are all four directions present, with no single
   direction exceeding 60% of the arrows?

### 3 — Special focus: Level 20 (crown)
- Verify the rendered board resembles a crown silhouette (spikes + jewel + base
  read as a crown).
- Check that arrow density (28 arrows per Phase 16) keeps the shape visually
  readable rather than cluttered or sparse.
- Confirm no collision or rendering defect specific to the crown geometry (the
  tapered spikes and flared base are the thinnest/densest regions).

### 4 — Rendering audit (all levels, especially 16–20)
Inspect the painter/hit-tester logic (and, if feasible, load levels in the UI):
- **Arrowhead overlap** — do scaled arrowheads (Phase 17) ever draw over a
  neighbouring arrow on the dense figure boards?
- **Node visibility at level start** — with coverage-based node alpha (Phase 17),
  are all nodes covered (near-invisible) at start, lighting up only as arrows
  escape?
- **Neon colors** — is the 5-color neon palette applied correctly?
- **Tap hit-testing** — is `hitSlop` (Phase 17 scaled) accurate on dense boards,
  i.e. a tap resolves to the intended arrow and not an adjacent one?

### 5 — Report findings BEFORE any code change
Produce a structured audit report:
- A per-level table: pass/fail for each check (2a–2h) across levels 16–20, plus
  the 1–15 regression summary from step 1.
- For every failure: the exact **arrow id(s)** and **coordinates**, the sweep
  path that exhibits the defect, and the bug class it matches (P14 gap-exit /
  P14.1 self-intersection / shape / disjointness / no-free-nodes / solvability /
  density / direction).
- A recommended fix per failure (prefer generator-side fixes in
  `tool/gen_levels.js`, consistent with Phases 14 / 14.1, over hand-editing JSON).

Only after this report — and approval — proceed to any fix.

### 6 — Fixes (only if the audit finds defects, and only after approval)
- Prefer fixing in the **generator** (`tool/gen_levels.js`) so the guarantee is
  structural and regressions are caught at generation time, not just at runtime.
- If the interior gap-exit check must be applied to figure levels, apply a
  **figure-aware** version (the naive bbox-relative check false-positives on
  concave silhouettes per Phase 16 — a corrected check must distinguish a real
  interior gap from a legitimate concavity on the shape's own edge).
- Add / extend `--validate-only` coverage so any new guarantee is enforced at
  generation time.
- Add / extend Dart tests for any runtime-observable defect found.
- Regenerate `assets/levels/manual_levels.json` only via the tool's
  write-only-if-valid path (`--generate` / `--generate-figures`); keep 1–15
  byte-identical unless a fix to those levels is explicitly justified.
